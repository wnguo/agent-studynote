# 🛠️ AI Agent 开发：手搓 vs 框架选型深度笔记

在工程实践中，框架（如 LangChain, LlamaIndex）与手搓（Hand-coding）的选择本质上是“开发效率”与“系统掌控权”的博弈。

---

## 1. 深度对比：框架 vs 手搓

| 维度 | 成熟框架 (Framework) | 自主手搓 (Hand-coding) |
| :--- | :--- | :--- |
| **上手速度** | 极快，内置大量样板代码 | 慢，需从零构建消息流 |
| **可观测性** | 差，抽象层极厚，堆栈追踪困难 | 极好，链路透明，断点直达 |
| **系统稳定性** | 受外部 Breaking Changes 影响 | 高，逻辑自持，不随第三方升级崩溃 |
| **性能优化** | 有通用性开销（冗余逻辑多） | 精简，只跑业务需要的代码 |
| **适用阶段** | POC、快速演示、简单的 RAG | 生产环境、高性能要求、高度定制化 |

---

## 2. 为什么生产环境更倾向“手搓”？

### A. 穿透“黑盒”的调试成本
框架的抽象层（Abstractions）在出问题时会变成障碍。手搓代码没有中间商，所有的 `messages` 拼接、`tool_calls` 解析都在明面上。

### B. 避免“过度设计”
框架为了兼容所有 LLM 和所有场景，设计得极其复杂。手搓可以针对特定业务场景（如：只需要支持 GPT-4 的 Function Call）做极致简化。

### C. 官方建议支持
Anthropic 等厂商明确建议：**“保持抽象层尽可能薄 (Keep your abstractions thin)”**。过早引入框架会让你离底层逻辑越来越远。

---

## 3. 最佳实践：渐进式架构

1. **初期 (Exploration)**：使用框架快速搭建 Demo。
2. **中期 (Production)**：将 Agent 的“心脏”（即 **ReAct Loop**、**消息列表维护**）重构为手写。
3. **长期 (Scale)**：仅将框架作为周边工具（如文档 Loader），核心控制流完全自主。

---

## 🎯 面试核心要点

- **不要只说框架好**：面试官更看重你对“技术债”和“可维护性”的思考。
- **提到“抽象陷阱”**：解释为什么 40 层的堆栈追踪对生产排障是灾难。
- **掌控感 (Control)**：强调在处理 Token 成本、响应延迟（Latency）和版本稳定性时，手搓带来的确定性。
