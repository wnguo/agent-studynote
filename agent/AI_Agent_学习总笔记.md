# AI Agent 学习总笔记

> 基于当前文件夹中的多篇 AI Agent 笔记整合而成，目标是形成一份结构完整、便于复习与面试表达的总笔记。

---

## 1. 什么是 AI Agent？

AI Agent（智能体）本质上是一个能够**围绕目标自主决策、调用工具、感知结果并持续调整**的 AI 系统。

它和传统 LLM 的核心区别，不是“能不能回答问题”，而是是否具备一个完整的**自主闭环**：

- 传统 LLM：偏向“一问一答”，主要负责文本生成
- AI Agent：偏向“给定目标后自主推进”，能够规划、执行、反思、迭代

一句话概括：

> **LLM 更像会回答问题的大脑，Agent 更像能主动完成任务的行动系统。**

---

## 2. AI Agent 的核心闭环

一个完整 Agent 的运行逻辑通常可以概括为：

**感知 → 规划 → 行动 → 观察 → 反思 → 再行动**

也可以理解为：

1. **感知**：接收用户目标，读取环境信息，必要时检索记忆
2. **规划**：判断接下来要做什么，是否要拆任务
3. **行动**：调用工具、执行代码、访问数据
4. **观察**：获取工具返回结果
5. **反思**：判断结果是否正确、是否还需要继续
6. **迭代**：进入下一轮循环，直到任务完成

这就是 Agent 与普通工作流或普通问答模型最大的区别：它不是一次性生成，而是**带反馈的连续决策系统**。

---

## 3. AI Agent 的四大核心组件

### 3.1 LLM：决策大脑

LLM 是 Agent 的推理与决策中心，负责理解目标、生成计划、选择工具、判断结果。

关键关注点：

- **推理能力**：决定任务拆解是否合理
- **上下文窗口**：决定一次能处理多少信息
- **结构化输出稳定性**：决定工具调用是否可靠
- **System Prompt**：定义 Agent 的角色、约束和行为方式

### 3.2 Planning：规划模块

规划模块负责把复杂目标拆成更容易执行的子任务。

常见作用：

- 任务拆分
- 顺序安排
- 失败后重规划
- 长链任务中的全局路径控制

没有规划时，模型容易“走一步看一步”；有规划时，模型更像先画地图再行动。

### 3.3 Memory：记忆系统

记忆系统决定 Agent 是一次性工具，还是持续进化的助手。

#### 短期记忆

用于保存当前任务相关信息，通常存在上下文窗口或会话内存中，例如：

- 对话历史
- 中间推理结果
- 工具观察结果
- 当前目标和约束

#### 长期记忆

用于持久化有价值的信息，通常结合向量数据库或结构化数据库实现，例如：

- 用户偏好
- 历史案例
- 业务知识
- 操作流程

#### 长期记忆常见分类

- **语义记忆**：事实、知识、规律
- **情节记忆**：发生过的事件与案例
- **程序记忆**：怎么做事的方法论或 SOP

### 3.4 Tools：工具系统

工具系统让 Agent 具备和外部世界交互的能力。

工具可以包括：

- 搜索
- 数据库查询
- 文件读写
- 调用 API
- 代码执行
- 邮件、消息、表单、浏览器操作等

核心原则是：

> **模型负责决策，代码负责执行。**

也就是说，LLM 不直接做体力活，而是告诉系统“该调哪个工具、用什么参数”。

---

## 4. Agent、Workflow、Tools 的区别

理解这三者的关键在于：

> **谁来决定“下一步该做什么”？**

### Tools

- 是最小能力单元
- 不具备自主决策能力
- 只能被调用后执行固定功能

### Workflow

- 是预先写好的流程编排
- 决策权在开发者手里
- 通常由代码中的条件分支控制路径

### Agent

- 是具备动态决策能力的系统
- 决策权在 LLM 手里
- 能根据环境反馈实时调整路径

### 一句话区分

- **Tool**：做具体动作
- **Workflow**：控制固定流程
- **Agent**：在不确定环境里自主决策

---

## 5. Workflow 与 Agent 的工程关系

实际生产中，很少纯粹依赖单一 Agent，也很少完全靠死板 Workflow。

更常见的是：

## **Agentic Workflow**

即：

- 用 **Workflow** 固定主流程和边界
- 用 **Agent** 处理需要灵活判断的节点
- 用 **Tools** 执行具体动作

这种设计兼顾：

- 可控性
- 可调试性
- 灵活性
- 成本控制

经验原则：

- 能写死的流程，不要交给 LLM
- 只有在存在不确定性时，才引入 Agent 决策

---

## 6. 三大核心范式

Agent 的工程设计里，最核心的三种范式是：

- **ReAct**
- **Plan-and-Execute**
- **Reflection**

### 6.1 ReAct：边想边做

ReAct = **Reasoning + Acting**

核心循环为：

**Thought → Action → Observation**

#### 含义

- **Thought**：先思考当前情况和下一步策略
- **Action**：调用工具或采取动作
- **Observation**：读取真实返回结果

#### 优点

- 简单直接
- 灵活性强
- 能根据外部结果实时修正

#### 缺点

- 容易越跑越长，Token 消耗高
- 缺少全局视野
- 长链任务容易迷路

#### 工程重点

- 设置最大迭代次数
- 明确结束条件
- 工具错误也必须回传给模型作为观察结果

适合场景：

- 中短任务
- 边界不清晰的问题
- 需要边探索边执行的流程

### 6.2 Plan-and-Execute：先规划再执行

这种范式把“想”和“做”拆开：

1. **Planner**：先做整体规划
2. **Executor**：逐步执行每个子任务
3. **Re-planner**：根据执行结果动态调整后续计划

#### 优点

- 具有全局视角
- 长链任务更稳
- 更容易并行化
- 可用强模型做规划、便宜模型做执行

#### 缺点

- 引入了额外规划成本
- 响应时间更长
- 对计划质量要求高

适合场景：

- 步骤多的复杂任务
- 有明显阶段划分的任务
- 需要可解释执行路径的任务

### 6.3 Reflection：执行后反思

Reflection 不是独立主流程，更像一个**质量增强器**。

典型闭环：

**生成 → 评估 → 改进**

#### 核心价值

- 发现逻辑错误
- 减少幻觉
- 提高输出质量
- 支持高标准任务交付

#### 常见形式

- 自我反思：模型自己检查自己
- Critic Agent：独立模型或独立 Agent 评估
- Reflexion：把反思结果沉淀成经验，供以后复用

#### 工程重点

- 必须有 PASS / FINISH 信号
- 必须限制最大反思次数
- 评估标准要可量化，不能只说“好不好”

适合场景：

- 代码生成
- 严肃文档
- 高质量总结
- 高风险任务的最终质检

## 7. 推理与规划能力演进

从规划角度看，常见方法包括：

### CoT（思维链）

让模型显式一步一步思考。

特点：

- 成本低
- 简单有效
- 是基础标配

### ToT（思维树）

在多个路径上探索、评估、剪枝。

特点：

- 质量更高
- 适合复杂决策
- 成本显著增加

### GoT（思维图）

让多个推理节点复用、合并与交叉连接。

特点：

- 更接近复杂问题求解
- 研究价值高
- 工程化难度大

## 8. 任务拆分：复杂任务的根基

任务拆分的核心思想是：

> **把大问题变成多个边界清晰的小问题。**

### 为什么要拆分？

- 降低上下文压力
- 提高输出稳定性
- 增强可控性
- 便于验收与重试
- 便于并行执行

### 两种典型拆分方式

#### 静态拆分

由开发者提前定义 SOP，适合固定业务流

#### 动态拆分

由模型根据目标实时生成计划，适合复杂开放任务

### 好的任务拆分标准

- **完备性**：覆盖原始需求
- **独立性**：步骤之间边界清晰
- **原子性**：输入输出明确，便于验收

### 进阶机制

- **Replan**：每执行一步后重评后续计划
- **递归拆分**：失败步骤继续向下拆

## 9. 记忆系统设计与压缩

随着会话变长，Context Window 会成为瓶颈，因此记忆管理不仅是“存”，更是“压缩、检索、筛选”。

### 9.1 记忆存储策略

#### 短期记忆

适合保存：

- 当前任务信息
- 最新几轮对话
- 中间结果
- 当前约束

#### 长期记忆

适合保存：

- 高价值结论
- 用户长期偏好
- 可复用经验
- 稳定知识

原则：

> **不要什么都存，只存长期有价值的东西。**

### 9.2 长期记忆的技术基础

典型实现方式：

- **Embedding**
- **向量数据库**
- **语义检索**
- 必要时结合关系型数据库存结构化事实

向量数据库适合：

- 模糊语义召回
- 历史经验检索
- 相似案例查找

关系型数据库适合：

- 用户画像
- 项目参数
- 订单、编号、配置等结构化信息

### 9.3 记忆压缩方法

#### 滑动窗口

只保留最近 N 条消息

优点：

- 简单
- 成本低

缺点：

- 容易丢失早期重要约定

#### 摘要压缩

把旧消息总结成简要文本再保留

优点：

- 平衡信息密度和成本

缺点：

- 会丢细节

#### 重要性过滤

给历史信息打分，只保留关键内容

优点：

- 控制噪音
- 提高有效信息密度

#### 结构化抽取

把对话转成 JSON、表结构或知识条目

优点：

- 检索效率高
- 特别适合垂直业务场景

缺点：

- 通用性较差
- 设计成本高

### 9.4 记忆管理的工程闭环

一个成熟的记忆流程通常是：

1. **Recall**：任务开始时召回相关记忆
2. **Inject**：把有价值记忆注入当前上下文
3. **Execute**：执行当前任务
4. **Consolidate**：任务结束后固化新经验

## 10. Multi-Agent：为什么要从单智能体走向多智能体？

当单 Agent 出现以下问题时，就该考虑 Multi-Agent：

- 上下文过长，记不住
- 一个 Prompt 很难兼顾多个角色
- 存在多个可并行的独立任务
- 需要更高专业分工
- 需要更高容错性和可维护性

一句话理解：

> **Multi-Agent 是用“分工”解决单 Agent 的记忆、角色和复杂度瓶颈。**

---

## 11. Multi-Agent 的典型协作模式

### 顺序式（Sequential）

多个 Agent 按固定顺序接力执行

适合：

- SOP 流程
- 明确前后依赖链路

### 层级式（Hierarchical）

由一个 Manager / Orchestrator 负责拆分、分发、汇总

适合：

- 复杂任务
- 需要统一调度和状态管理的系统

### 协作式 / 黑板式（Joint / Blackboard）

多个 Agent 共享状态，共同处理一个目标

适合：

- 创意任务
- 开放式探索
- 高并发信息收集

---

## 12. Single-Agent vs Multi-Agent 选型原则

### 先用 Single-Agent 的情况

- 任务规模不大
- 路径相对明确
- 不需要并行
- 单模型上下文足够
- 系统需要尽量简单

### 升级为 Multi-Agent 的信号

1. **记忆天花板**：上下文装不下
2. **能力冲突**：一个 Agent 同时扮演多个矛盾角色
3. **性能瓶颈**：存在大量可并行的独立子任务

### 选型建议

- 默认优先从 **Single-Agent** 起步
- 遇到明确瓶颈再升级为 **Multi-Agent**
- 生产环境优先考虑**中心化编排**
- 去中心化方案更适合研究，不适合工程落地

---

## 13. Multi-Agent 的工程关键点

### 13.1 Orchestrator 的价值

Orchestrator 不只是派单员，更是：

- 状态协调者
- 上下文守门人
- 结果整合者
- 失败恢复控制器

### 13.2 动态切换机制

常见方式：

- **静态路由**：基于规则或分类器选择 Agent
- **动态编排**：由主 Agent 根据状态决定下一节点
- **条件退出**：设置完成信号、失败信号和停止条件

### 13.3 常见挑战

- 上下文爆炸
- 多 Agent 相互推责
- 状态不一致
- 通信成本高
- Token 成本上升

### 对应对策

- 摘要和选择性上下文传递
- 统一状态中心
- 最大循环次数限制
- 明确完成标准
- 只对必要任务使用多 Agent

## 14. 手搓 vs 框架：如何选型？

AI Agent 工程里，框架和手搓的矛盾，本质是：

> **开发效率 vs 系统掌控权**

### 框架的优势

- 上手快
- 样板代码少
- 适合 POC、Demo、快速验证

### 框架的劣势

- 抽象层太厚
- 调试困难
- 升级风险高
- 性能和控制不够精细

### 手搓的优势

- 透明可控
- 便于调试
- 便于优化 Token、延迟和状态流
- 更适合生产环境

### 手搓的劣势

- 开发初期慢
- 对工程能力要求更高

### 最佳实践

- **初期**：用框架快速验证
- **中期**：把核心控制流手写
- **长期**：仅保留框架做外围工具，核心逻辑自己掌控

核心建议：

> **保持抽象层尽可能薄。**

## 15. AI Agent 的工程化最佳实践

### 15.1 从简到繁

- 先做最小可运行版本
- 先用 ReAct 跑通闭环
- 再引入 Planning、Memory、Reflection
- 最后根据瓶颈考虑 Multi-Agent

### 15.2 能写死就别交给模型

例如：

- 数据清洗
- 参数校验
- 状态机流转
- 格式转换
- 固定 SOP

这些更适合 Workflow / 普通代码，不适合交给 LLM 决策

### 15.3 所有增强能力都有代价

引入以下能力时，都要意识到成本：

- Planning：增加推理成本
- Reflection：增加时延和 Token
- Memory：增加存储和检索复杂度
- Multi-Agent：增加通信和调度成本

### 15.4 关键系统必须有硬边界

必须设计：

- 最大循环次数
- 最大反思轮次
- 明确完成信号
- 错误回传机制
- 状态一致性约束

## 16. 面试/复习高频总结

### 16.1 什么是 Agent？

Agent 是能围绕目标自主执行闭环任务的 AI 系统，区别于普通 LLM 的关键在于自主性和反馈循环。

### 16.2 Agent 的核心组件是什么？

- LLM
- Planning
- Memory
- Tools

### 16.3 ReAct、Plan-and-Execute、Reflection 的区别？

- ReAct：边想边做，灵活
- Plan-and-Execute：先计划后执行，适合长任务
- Reflection：执行后质检，提升输出质量

### 16.4 Workflow 和 Agent 的区别？

关键在于谁决定下一步：

- Workflow：开发者决定
- Agent：LLM 决定

### 16.5 为什么需要 Memory？

因为没有记忆，Agent 只能完成单次任务；有了记忆，Agent 才可能成为连续进化的助手。

### 16.6 为什么需要 Multi-Agent？

因为单 Agent 会遇到上下文、角色冲突、并行效率等瓶颈，多 Agent 可以通过分工与隔离提升系统表现。

### 16.7 为什么生产环境推崇 Agentic Workflow？

因为它在“可控”与“灵活”之间取得了最实用的平衡。

---

## 17. 一套推荐的认知框架

如果只用一句话总结 AI Agent 的整体设计思路，可以这样记：

> **用 Workflow 保证确定性，用 Agent 处理不确定性，用 Tools 执行动作，用 Memory 保持连续性，用 Reflection 保证质量，用 Multi-Agent 解决复杂度上限。**

---

## 18. 总结

AI Agent 并不是“给 LLM 加几个工具”这么简单，它本质上是一套围绕目标、自主决策、可持续迭代的系统设计方法。

从工程视角看，真正重要的不是概念堆砌，而是以下几个问题：

- 谁在决策？
- 决策是否可控？
- 状态如何保存？
- 错误如何纠正？
- 成本如何约束？
- 什么时候该从单体升级到多体？

真正成熟的 Agent 系统，通常不是某一种范式的极致，而是多种机制的组合：

- 主流程靠 Workflow 保底
- 局部灵活性靠 Agent 补足
- 长任务靠 Planning 管理
- 高质量靠 Reflection 兜底
- 长期能力靠 Memory 沉淀
- 高复杂度靠 Multi-Agent 分治
