# 🏗️ Agent 设计范式与 Workflow 深度笔记

理解 Agent 工程化的核心：在**灵活性**（Agent）与**确定性**（Workflow）之间寻找平衡。

---

## 1. Workflow vs Agent：本质区别

| 维度 | Workflow (工作流) | Agent (智能体) |
| :--- | :--- | :--- |
| **决策者** | **开发者**（提前写死逻辑） | **LLM**（运行时动态决策） |
| **控制流** | 顺序/分支执行：A -> B -> C | 循环迭代：感知 -> 规划 -> 行动 |
| **确定性** | **高**（行为可预测，好调试） | **低**（同样的输入路径可能不同） |
| **灵活性** | 低（遇到意外情况容易报错） | **高**（能处理模糊、非结构化任务） |
| **适用场景** | 流程固定的业务系统 | 路径不确定的复杂研究任务 |

---

## 2. 三大核心 Agent 设计范式

### ① ReAct (Reasoning + Acting) —— “边想边做”
* **核心：** 每一轮循环由 `Thought` -> `Action` -> `Observation` 组成。
* **价值：** 通过让模型先写出推理过程（Thought），减少错误决策，提高逻辑透明度。
* **局限：** 视野局限在“下一步”，处理长任务时容易迷失整体目标。

### ② Plan-and-Execute —— “先谋后动”
* **核心：** 规划器先拆解任务列表，执行器按表干活，并根据结果进行**动态重规划**。
* **价值：** 具备全局视野，适合步骤极多、逻辑链条长的复杂任务。
* **代价：** 增加了规划环节的 Token 消耗和响应延迟。

### ③ Reflection (反思) —— “自我纠错”
* **核心：** 任务完成后增加一步评估。若质量不达标，模型分析失败原因（错题本）并重试。
* **代表作：** **Reflexion**。
* **价值：** 极大幅度提升代码、写作等高标准任务的成功率。

---

## 3. 工程化最佳实践：Agentic Workflow

生产环境最成熟的方案：**整体 Workflow + 局部 Agent**。
* **原则：** 能用代码写死的流程（如数据清洗、格式转换）绝对不要交给 LLM 决策。
* **优势：** 保证主链路 100% 可控，仅在需要“自主探索”的节点开启 Agent 模式。

---

## 🎯 面试核心要点总结

1. **谁决定下一步？** 区分 Workflow 和 Agent 的唯一标准。
2. **代价是什么？** 所有的智能提升（Plan、Reflection）都伴随着 **Token 成本** 和 **响应延迟** 的增加。
3. **架构建议：** 遵循“从简到繁”的原则，先构建确定性的 Workflow，再按需引入 Agent 范式。
