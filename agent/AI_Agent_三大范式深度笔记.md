# 🏆 Agent 三大范式：ReAct, Plan-and-Execute, Reflection 深度笔记

理解这三种范式，本质上是理解如何在**灵活性**、**控制力**与**质量**之间进行权衡。

---

## 1. 核心范式对比：解决什么问题？

| 范式名称 | 做事风格 | 核心解决痛点 | 典型类比 |
| :--- | :--- | :--- | :--- |
| **ReAct** | 边想边做，实时调整 | 单步决策的逻辑性与实时纠错 | **外卖骑手**：根据路况随时改道 |
| **Plan-and-Execute** | 先定计划，分步执行 | 长流程任务容易“跑偏”或丢失目标 | **项目经理**：先出方案再分头干活 |
| **Reflection** | 执行之后，自我检查 | 输出质量不达标、逻辑或事实错误 | **学生检查**：交卷前回头找错题 |

---

## 2. 详细特性拆解

### ① ReAct (推理+行动)
* **本质**：将“思考”与“观察外部结果”交替进行。
* **优点**：简单、极其灵活，适合处理边界不清晰的简单任务。
* **缺点**：随着步骤增多，Context 会迅速膨胀，且缺乏全局视野。

### ② Plan-and-Execute (规划与执行)
* **本质**：将“想”和“干”完全解耦。
* **优势**：
    1. **结构化**：任务链路清晰可预测。
    2. **降本增效**：可以用强模型（如 GPT-4）做规划，用极便宜的小模型做具体的步骤执行。
* **进阶**：**动态 Replan**（每步执行完评估是否需要修改后续计划）。

### ③ Reflection (反思增强)
* **定位**：这不是一个独立的流程，而是给前两者的**检查 Buff**。
* **关键闭环**：`生成` -> `评估` -> `反思建议` -> `改进重做`。
* **代价**：Token 消耗和响应延迟会显著增加。

---

## 3. 工程化选型建议 (核心心法)

1. **按需升级**：不要一上来就搞全套。先用 **ReAct** 跑通流程。
2. **长任务选 Plan**：如果发现 ReAct 在超过 5 步的任务中经常胡言乱语，立刻切换到 **Plan-and-Execute**。
3. **高质量选 Reflection**：如果任务要求代码必对、公文必严谨，必须在关键节点加上 **Reflection** 环节。
4. **混合架构**：生产环境下最强方案是——用 **Plan-and-Execute** 框住大流程，在每个执行节点用 **ReAct** 处理小循环，最后加一步 **Reflection** 质检。

---

## 🎯 面试高频考点

- **为什么说 Reflection 不是独立范式？** 因为它必须依附于一个执行逻辑，是针对输出质量的二次优化。
- **解耦有什么好处？** 方便针对性优化模型、方便并行执行无依赖的步骤、显著降低长任务的 Token 总消耗。
- **如何防止反思死循环？** 必须设置明确的 `Max_Reflections`（最大反思轮次）和清晰的评分阈值。
