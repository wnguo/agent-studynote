# 🏗️ Single-Agent 与 Multi-Agent 设计选型深度笔记

在构建 AI Agent 系统时，架构的选择决定了系统的上限与维护成本。

---

## 1. 核心方案对比

| 维度 | Single-Agent (单智能体) | Multi-Agent (中心化编排) | Multi-Agent (去中心化) |
| :--- | :--- | :--- | :--- |
| **角色定义** | 一个人包揽所有活 | 一个指挥官 + 多个执行者 | 多个 Agent 自行协商 |
| **Context 压力** | 极高，容易导致“失忆” | 低，每个 Worker 环境独立 | 低 |
| **专业性** | 泛才（什么都懂一点） | 专才（每个 Worker 专注一块） | 专才 |
| **可控性** | 极高，链路唯一 | 高，调度逻辑清晰 | 极低，行为难以预测 |
| **工程实用性** | **高**（适合中小型任务） | **极高**（生产环境首选） | **低**（多见于学术研究） |

---

## 2. 选型三原则 (Decision Matrix)

当出现以下情况时，必须从 Single-Agent 升级为 Multi-Agent：
1. **记忆天花板**：任务中间产出太多，单次 Context 窗口装不下。
2. **能力冲突**：同一个 Prompt 很难让模型既做“严谨的财务审计”又做“发散的文案创意”。
3. **性能瓶颈**：任务链路中存在多个互不依赖的子任务（如：同时调研三家竞品），需要并行执行以降低延迟。

---

## 3. 中心化编排 (Orchestrator) 的执行流

1. **目标拆解**：指挥官将 User Prompt 拆分为 Task A, B, C。
2. **任务指派**：根据 Agent 的描述（Description）将 Task A 给 Researcher，Task B 给 Coder。
3. **状态同步**：指挥官维护一个全局状态，确保 B 拿到 A 的输出。
4. **结果汇总**：指挥官对所有 Worker 的结果进行最后的逻辑校验与整合。

---

## 🎯 面试核心要点

- **不要盲目求多**：强调“渐进式演进”。先做 Single-Agent，遇到瓶颈再拆分，避免过度设计（Over-engineering）。
- **去中心化的弊端**：在面试中一定要提到去中心化方案在工程上的不可控性（无失败感知、无执行顺序保障、无完成标准）。
- **Orchestrator 的价值**：它不仅是分配任务，更是**上下文的守门人**，它确保发给每个 Worker 的信息都是高度精炼且相关的。
