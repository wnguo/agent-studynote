# 🧠 AI Agent 推理模式：ReAct 深度笔记

ReAct (Reasoning + Acting) 是 AI Agent 最经典的设计范式，由 Google 和普林斯顿大学提出。它实现了“逻辑推理”与“工具调用”的交替闭环。

---

## 1. 核心公式：T + A + O

ReAct 的运行逻辑是一个周而复始的循环，每一轮由三个核心环节组成：

- **Thought (推理/思考)**：
  - 模型在执行前先输出一段文字，分析当前进度，思考接下来该干什么。
  - **作用**：为模型提供“推理空间”，减少冲动决策。
- **Action (行动/工具调用)**：
  - 模型根据 Thought 的结论，决定调用哪个工具、传什么参数。
  - **格式**：通常输出结构化的 JSON。
- **Observation (观察/反馈)**：
  - 工具执行后的真实结果被反馈给模型。
  - **作用**：让模型感知外部环境，纠正预期的偏差。

---

## 2. 为什么 ReAct 是必选范式？

| 模式 | 特点 | 缺陷 |
| :--- | :--- | :--- |
| **纯推理 (CoT)** | 只有思考过程，没有外部行动 | 知识受限于训练数据（知识冻结），无法联网 |
| **纯行动 (Acting)** | 只调工具，不写推理过程 | 逻辑容易断裂，在复杂任务中幻觉严重，选错参数 |
| **ReAct** | **推理与行动交替进行** | **最均衡方案**。能根据工具返回的结果实时调整策略。 |

---

## 3. 工程化核心要点

在实际开发 Agent 时，ReAct 模式需要处理以下问题：

1. **死循环与退出**：
   - 必须设置 `max_iterations`（最大迭代次数），防止模型在两个工具间反复横跳。
   - 明确定义 `Final Answer` 标记，告知模型任务已完成。
2. **错误处理**：
   - 如果工具调用报错，报错信息必须作为 `Observation` 给回模型。
   - 强的模型（如 GPT-4, Claude 3.5）通常能根据报错信息自我修复参数。
3. **Prompt 模板设计**：
   - 在 Prompt 中提供少量的 ReAct 样例（Few-shot），引导模型严格遵守 `Thought -> Action -> Observation` 的格式。

---

## 4. ReAct 的局限性

- **Token 消耗**：反复的循环会导致 Prompt 越来越长。
- **局部视野**：ReAct 属于“走一步看一步”，缺乏全局规划能力。对于极其复杂的长链任务，建议配合 **Plan-and-Execute** 范式。
