# 🛠️ Workflow, Agent, Tools 深度辨析笔记

理解这三者的核心在于一个关键维度：**谁来做“下一步该干什么”的决策？**

---

## 1. 三层架构详解

### 第一层：Tools (工具) —— “最小积木”
* **定义**：封装好的单一功能函数（如：搜索、发邮件、读数据库）。
* **决策能力**：**无**。它像瑞士军刀的刀片，不被“手”翻开就永远不动。
* **特征**：被动调用、职责单一、输出明确。

### 第二层：Agent (智能体) —— “自主决策者”
* **定义**：拿着工具、以 LLM 为大脑的决策系统。
* **决策能力**：**有 (自主动态决策)**。LLM 自行决定“调哪个工具、要不要继续、何时结束”。
* **核心循环**：`思考 (Thought)` -> `行动 (Action)` -> `观察 (Observation)` 的反复迭代。

### 第三层：Workflow (工作流) —— “编排总指挥”
* **定义**：将 Agent、LLM、Tools 按照预设逻辑组织起来的流程框架。
* **决策能力**：**无 (开发者预设)**。流程的走向由程序员写的 `if/else` 代码死死卡住。
* **特征**：高确定性、可预测、易于调试。

---

## 2. 核心维度对比表

| 维度 | Tools (工具) | Agent (智能体) | Workflow (工作流) |
| :--- | :--- | :--- | :--- |
| **决策者** | 无（只执行） | **LLM 实时动态决策** | **开发者（提前写死）** |
| **确定性** | 极高 | 低（路径可能飘忽） | 高（完全可预测） |
| **灵活性** | 极低 | **极高（应对未知情况）** | 较低（难以动态调整） |
| **适用场景** | 封装具体的单一原子能力 | 处理路径未知的复杂任务 | 流程固定的业务系统 |

---

## 3. 生产环境的主流：Agentic Workflow

在实际工程中，纯 Agent 太难控，纯 Workflow 太死板。目前的“工业标准”是 **Agentic Workflow**：
> **用 Workflow 固定主流程的骨架，在需要灵活判断的关键节点嵌入 Agent。**

### 常见的编排模式：
1. **Prompt Chaining (提示链)**：流水线串联步骤。
2. **Routing (路由)**：先分类，再分发到不同分支。
3. **Evaluator-Optimizer (评估-优化)**：生成 -> 评估 -> 反馈修改 -> 重新生成。
4. **Orchestrator-Workers (编排-工人)**：主节点拆解任务，分发给子节点执行。

---

## 🎯 面试必记总结

1. **分清节点与流程**：Workflow 的节点可以是 Agent，但控制流是代码写的。
2. **决策权的归属**：Agent 掌握决策权（开发者给目标）；Workflow 掌握决策权（开发者写路径）。
3. **确定性 vs 灵活性**：生产环境追求平衡，Agentic Workflow 兼顾了“好调试、成本可控”和“能处理复杂变化”。
