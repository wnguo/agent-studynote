# 🤖 Multi-Agent (多智能体) 系统深度笔记

Multi-Agent 系统是解决复杂长流程任务、打破单个模型能力上限的关键方案。

---

## 1. 核心价值：为什么要多 Agent 协作？

单个 Agent 存在两个无法通过简单优化绕过的“死穴”：
- **Context 崩溃**：任务越复杂，带入的上下文越多。超出 Token 限制后，模型会丢失关键指令或产生幻觉。
- **干扰与噪音**：让模型同时处理不同职能（如写代码与做产品分析），信息会互相干扰。
- **解决方案**：Multi-Agent 通过“分而治之”，让每个 Agent 拥有独立的、干净的上下文环境。

---

## 2. 协作架构模式

1. **Sequential (顺序模式)**：
   - 流程：Agent A -> Agent B -> Agent C。
   - 场景：固定 SOP 流程，如：文章撰写 -> 自动翻译 -> 格式校对。
2. **Hierarchical (层级/中心化模式)**：
   - 流程：有一个 Manager Agent 负责拆解任务并分发给 Workers。
   - 场景：复杂多变的动态任务。
3. **Joint (并行/协作模式)**：
   - 流程：多个 Agent 同时对一个任务进行不同维度的处理（如：并行搜集不同竞品信息）。
   - 场景：追求执行效率。

---

## 3. 核心优势对比

- **专业分工**：每个 Agent 可以有不同的 System Prompt、不同的工具集，甚至不同的基座模型（如：规划用 GPT-4，简单执行用 7B 模型）。
- **并行加速**：识别任务依赖，将无依赖的任务并行化，大幅降低端到端延迟。
- **容错性**：单个 Agent 失败不影响全局，可以针对特定节点进行重试或反思（Reflection）。

---

## 4. 生产环境建议

- **框架选型**：
    - **CrewAI**：适合角色扮演和流程编排，简单易上手。
    - **LangGraph**：适合需要精细控制循环逻辑、状态管理的复杂场景。
    - **Microsoft Agent Framework (MAF)**：微软推出的原 AutoGen 升级版，面向生产级应用。
- **避坑指南**：Agent 越多，通信成本和 Token 开销越大。如果 Single-Agent + 任务拆分能解决，不要盲目上 Multi-Agent。

---

## 🎯 面试核心要点

- **两个硬限制**：Context Window 上限、角色干扰（噪音）。
- **并行化**：这是 Multi-Agent 在工程上带来的最直接收益。
- **隔离性**：强调“关注点分离”，这使得系统更容易调试和维护。
