# 🧩 AI Agent 任务拆分深度笔记

任务拆分是复杂 Agent 系统的基石，其核心思想是“化繁为简，分而治之”。

---

## 1. 为什么要进行任务拆分？

- **上下文管理**：避免 LLM 的上下文窗口（Context Window）中堆积过多的中间状态，防止模型因“信息过载”而产生的幻觉或逻辑混乱。
- **可控性与确定性**：每一个小的步骤（原子操作）都更容易编写验收标准（断言），方便进行局部纠错和自动重试。
- **注意力聚焦**：让 LLM 在同一时间只处理一个子目标，产出的深度和质量显著高于“一口气完成”的效果。

---

## 2. 任务拆分的方法论

### ① 静态拆分 (Workflow)
- **定义**：由开发者提前写死固定的 SOP（标准作业程序）。
- **场景**：业务逻辑高度标准化的场景（如：自动化客服受理流程）。

### ② 动态拆分 (Plan-and-Execute)
- **定义**：由 LLM 作为“规划者”，根据任务目标实时生成执行清单。
- **三个阶段**：
  1. **规划 (Plan)**：梳理逻辑链条，确定先后顺序。
  2. **执行 (Execute)**：分工给具体的工具或子 Agent。
  3. **汇总 (Summarize)**：整合多点产出，完成连贯性收口。
- **进阶：自适应拆分**。如果执行失败，递归地将该失败步骤进一步拆解。

---

## 3. 性能优化：并行化处理

- **串行 vs 并行**：
  - 传统做法：步骤 1 -> 步骤 2 -> 步骤 3（总时长为 1+2+3）。
  - 优化做法：通过**依赖关系分析**，将互不干扰的步骤（如：并发搜索三个数据源）使用 `asyncio.gather` 同时启动。
- **收益**：端到端延迟通常可降低 **40%-60%**。

---

## 4. 好的任务拆分标准

- **完备性**：所有步骤加总后，完整覆盖了原始 Prompt 的需求，无遗漏。
- **独立性**：各步骤职责边界清晰，不存在“你中有我、我中有你”的重合。
- **原子性**：单一步骤应尽可能是一个“原子操作”，即输入输出明确且单一。

---

## 🎯 面试要点总结

- **核心价值**：提升模型输出的稳定性，降低 Token 成本（分段处理不需要带全程历史），优化响应速度（并行化）。
- **粒度控制**：拆得太细会增加通讯成本，拆得太粗会降低质量。理想的粒度是“能为一个步骤写出明确的验收标准”。
- **进阶机制**：提到 **Replan (动态重规划)** 机制，即每执行完一步都重新评估后续计划的合理性，是高水平的表现。
