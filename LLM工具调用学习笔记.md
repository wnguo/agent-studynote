## 一、这部分内容到底在学什么

这套内容表面上是在讲“大模型怎么调用工具”，但真正学的是一条完整链路：

- 模型如何表达调用意图
- 模型为什么能学会工具调用
- 工具为什么需要标准化接入
- 多 Agent 场景下如何做能力协作
- 真正上线后如何做统一治理

所以这不是一个点，而是一整套分层体系。

我把它整理成下面这条主线：

```text
Function Calling
→ 工具调用训练
→ MCP
→ A2A / Skill
→ LLM Gateway
```

如果按学习顺序理解，就是：

1. 先理解一次工具调用是怎么发生的
2. 再理解模型为什么会这种能力
3. 再理解工具如何标准化接入
4. 再理解多 Agent 如何协作
5. 最后理解系统上线后的治理问题

---

## 二、先建立整体分层认知

我觉得这部分最容易乱，是因为不同概念不在一个层次上。

可以先把它们拆成三层：

### 1. 单次调用层：Function Calling

解决的问题是：

> 模型如何把“我要调工具”这件事用结构化方式表达出来。

### 2. 工具接入层：MCP

解决的问题是：

> 工具如何被统一暴露、统一接入、统一发现。

### 3. 协作与调度层：Skill / A2A

解决的问题是：

> 多个 Agent 存在时，任务该交给谁，以及 Agent 之间如何协作。

再往外一层，就是工程治理：

### 4. 生产治理层：LLM Gateway

解决的问题是：

> 多模型、多团队、多 Key、多成本场景下，如何统一管理和控制。

我自己的记忆方式：

- `Function Calling`：怎么调
- `MCP`：怎么接
- `Skill / A2A`：谁来做
- `Gateway`：怎么管

## 三、Function Calling 学习笔记

### 1. Function Calling 的本质

Function Calling 不是让模型自己执行工具，而是让模型输出一段 **结构化调用请求**。

真正的执行者仍然是宿主程序。

也就是说，流程是：

1. 开发者把工具描述成 `schema`
2. 模型读取工具定义和用户问题
3. 模型决定是否调用工具
4. 如果调用，就输出结构化参数
5. 程序解析后真正执行函数 / API
6. 执行结果再喂回模型
7. 模型给出最终回答

这里一定要记住一句话：

> 模型负责“决定怎么调”，程序负责“真正去调”。

### 2. 为什么它重要

如果没有 Function Calling，很多系统就只能靠自然语言猜意图：

- 模型说“我查一下天气”
- 后端再猜它是不是想调天气接口

这种做法非常不稳定。

Function Calling 的价值在于，它把“调用工具”从模糊表达变成了结构化协议。

### 3. 我对它的理解

我会把 Function Calling 看成是 **LLM 与外部世界之间的最小连接单元**。

因为模型本身只会生成 token，它碰不到数据库、天气接口、浏览器、文件系统。  
它要接入现实世界，必须靠一个中间层把意图结构化，再由程序去执行。

这个最小连接单元，就是 Function Calling。

### 4. 容易答错的点

- 误以为模型会直接执行工具
- 把工具调用理解成普通文本输出
- 不知道 `schema` 的作用
- 不知道执行结果要回传给模型
- 不知道为什么通常是“两阶段过程”

## 四、大模型是怎么学会工具调用的

这部分是我觉得最值得单独记下来的内容。

### 1. 预训练不会天然教会工具调用

大模型在预训练阶段，本质目标还是“预测下一个 token”。

这能让模型学会很多语言能力，比如：

- 语言理解
- 文本续写
- 推理归纳
- 风格模仿

但它不会天然学会：

- 什么时候该调工具
- 输出怎样的结构化 JSON
- 参数如何严格对齐 schema

所以工具调用能力不是预训练自带的，而是后续训练专门教出来的。

### 2. SFT 教会的是“怎么调”

监督微调会给模型很多“标准示例”：

- 用户问了什么
- 当前有哪些工具
- 应该调用哪个工具
- 参数怎么填
- 工具返回什么
- 最终答案怎么组织

通过这些样本，模型学会：

- 识别调用时机
- 按规范输出工具调用
- 从问题里提取参数
- 完成“调用 → 返回 → 总结”的链路

我对这一步的记忆是：

> `SFT = 教模型学会流程和格式`

### 3. RLHF / RLAIF 教会的是“该不该调”

只学会格式还不够，模型还要学会判断边界。

比如：

- `1+1 等于几`，没必要调工具
- `今天上海天气`，更适合调实时工具

这就要求模型不仅“能调”，还要“会判断值不值得调”。

所以偏好优化阶段更像是在教模型：

- 不要乱调
- 不要过度调用
- 能直接回答就直接回答
- 真需要外部信息时再调工具

我的总结是：

- `SFT` 解决“会不会调”
- `RLHF / RLAIF` 解决“该不该调”

### 4. 一句学习结论

> 工具调用能力是后天训练出来的，不是预训练自然长出来的。

## 五、MCP 学习笔记

> 这一部分我按小林专题的整体脉络做理解性整理，重点抓“它为什么存在、和 Function Calling 的边界是什么”。

### 1. 为什么会需要 MCP

当工具越来越多时，工程上会有很多重复问题：

- 每个工具接入方式不同
- 每个 AI 应用都要各接一遍
- 工具发现方式不统一
- 维护和复用成本越来越高

所以需要一个更标准化的工具接入方式。

MCP 可以理解成：

> 面向 AI 应用和工具服务之间的一种标准化接入协议。

### 2. 它和 Function Calling 的区别

我觉得最清晰的理解方式是：

- `Function Calling` 关注的是“这次该怎么调”
- `MCP` 关注的是“工具整体怎么被标准化暴露”

也就是：

- 前者偏单次调用
- 后者偏工具生态接入

### 3. 为什么不要把它们当成替代关系

因为两者处理的问题根本不在一个层次。

一个系统完全可能同时存在：

- 用 `MCP` 接入一批标准化工具
- 再通过 `Function Calling` 在运行时决定调用哪个工具

所以我的结论是：

> MCP 不一定替代 Function Calling，它更像是把工具管理这件事前置并标准化了。

### 4. 关于通信方式怎么理解

专题里把 `SSE`、`WebSocket`、`WebRTC` 放在 MCP 相关内容里讲，我的理解是想说明：

- 工具系统不是只要“能调”就够
- 还涉及消息如何传输、如何实时更新、如何持续交互

先记最核心区别就够：

- `SSE`：服务端推送，适合流式消息
- `WebSocket`：双向长连接，适合持续交互
- `WebRTC`：更偏低延迟实时媒体或复杂实时通信

这一段不一定要死背协议细节，但要理解它在 AI 系统中的位置。

## 六、Skill 与 A2A 学习笔记

### 1. Skill 是什么

我觉得 Skill 最容易被误解成“另一个函数”，但其实不是。

Skill 更像：

> Agent 对外声明的能力标签或任务能力边界。

比如：

- `get_weather(city)` 是函数
- “数据分析”“生成周报”“代码审查”更像 Skill

也就是说：

- 函数更偏原子动作
- Skill 更偏任务能力

### 2. 为什么 Skill 不是 Function

因为一个 Skill 背后可能包含很多步骤：

- 调多个函数
- 做多轮推理
- 查询多个数据源
- 调用其他 Agent

所以 Skill 是更高层的抽象。

### 3. A2A 的位置

A2A 更关注 Agent 之间怎么协作。

如果说 Function Calling 是“模型调用工具”，那么 A2A 更像：

> 一个 Agent 把任务交给另一个 Agent 去处理。

所以它解决的问题不再是单次函数执行，而是任务协同。

### 4. 我的记忆方式

我会这样记：

- `Function`：做动作
- `Skill`：表能力
- `A2A`：做协作

## 七、Function Calling、MCP、Skill 的关系

这是整套内容里最适合用来做总结的一题。

### 1. 一句话关系

> Function Calling 负责单次工具调用，MCP 负责工具标准化接入，Skill 负责能力声明与任务分发。

### 2. 按层次理解

| 概念 | 关注点 | 典型问题 |
| --- | --- | --- |
| Function Calling | 单次调用 | 这次该调哪个函数、参数是什么 |
| MCP | 工具接入 | 这些工具怎样统一暴露和发现 |
| Skill | 能力声明 | 这个 Agent 擅长处理什么任务 |

### 3. 我自己的最终理解

这三者不是竞品，而是分层协作关系。

如果把 AI 系统类比成公司协作：

- Function Calling 像“某个员工去调用一个具体工具”
- Skill 像“员工的岗位职责说明”
- MCP 像“公司内部统一的系统接入标准”

这么想会很顺。

## 八、LLM 网关学习笔记

### 1. 为什么会有 LLM 网关

当系统真正进入生产环境后，问题就不只是“能不能调模型”了，而是：

- 模型供应商不止一个
- API Key 不止一套
- 团队不止一个
- 成本也必须统计和控制

这时就需要一个统一治理层。

### 2. 它解决什么问题

我总结下来，LLM 网关主要解决六类问题：

- **统一入口**：业务只接一个接口
- **模型路由**：底层可以切不同模型供应商
- **Key 管理**：统一保管与分发密钥
- **限流配额**：避免资源被打爆
- **成本统计**：统计 token 和费用
- **安全治理**：做审计、过滤、权限控制

### 3. 为什么它不只是“反向代理”

普通反向代理更偏流量转发。  
LLM 网关除了转发，还会承担：

- 模型抽象
- 成本治理
- 安全策略
- 观测与审计
- 缓存与优化

所以它更像“AI 调用治理平台”。

### 4. 我对这一部分的感受

这一题特别像“从 demo 到生产”的分水岭。

前面在讨论：

- 模型怎么调工具

到了这里，开始讨论：

- 系统怎么上线
- 团队怎么协作
- 成本怎么管
- 风险怎么控

## 九、这套知识最值得记住的结论

### 1. 核心判断句

- 模型不会直接执行工具，它只是输出调用请求
- 工具调用依赖结构化 schema，而不是自然语言猜测
- 工具调用能力主要来自后训练，而不是预训练天然具备
- SFT 解决“怎么调”，偏好优化解决“该不该调”
- MCP 关注工具标准化接入，不等于 Function Calling
- Skill 是能力声明，不是函数本身
- A2A 处理的是 Agent 之间协作
- LLM 网关处理的是生产治理问题

### 2. 最适合面试时的总括表达

如果让我用一段话总结这套内容，我会这样说：

> 我会把 LLM 工具调用分成四层来看。最底层是 Function Calling，它负责让模型输出结构化工具调用请求；再上一层是 MCP，它负责把工具标准化接入和暴露出来；再上一层是 Skill 和 A2A，它们解决多 Agent 场景下的能力声明和任务协作；真正上线后，还需要 LLM 网关去做统一接入、限流、成本、安全和审计治理。这样整套体系才完整。

## 十、我建议的复习顺序

如果后面要复习，我会按这个顺序重新过一遍：

1. 先复习 `Function Calling` 原理
2. 再复习“工具调用能力怎么训练出来”
3. 再看 `MCP` 和通信方式
4. 再看 `Skill`、`A2A`
5. 最后看 `LLM Gateway`

如果是为了面试背诵，我会优先背下面三组：

- **边界**：模型负责什么，程序负责什么
- **分层**：FC / MCP / Skill 各解决什么
- **工程化**：Gateway 为什么一定会出现
