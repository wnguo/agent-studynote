## 二、Function Calling 基础篇

### 1. 什么是 Function Calling？讲讲它的原理？

#### 核心问题

这一篇在回答：

> 大模型怎么从“只会生成文本”变成“能够触发外部工具执行”？

#### 我的笔记

- Function Calling 本质上是一种 **结构化工具调用机制**
- 开发者先把工具定义成 `schema`
- 模型根据用户问题判断要不要调工具
- 如果要调，模型输出工具名和参数
- 宿主程序再真正执行函数或外部 API
- 工具返回结果后，再交回模型总结成自然语言

最重要的一句：

> **模型不执行工具，模型只是输出调用请求；真正执行的是宿主程序。**

#### 我觉得这篇最该记住的点

- `schema` 是工具的描述
- `tool_calls` 是模型的结构化输出
- 工具执行一定发生在模型外部
- 整个过程通常是“两轮模型调用 + 一次中间执行”

#### 面试回答关键词

`schema`、结构化输出、宿主程序执行、结果回传、两阶段调用

### 2. LLM 是如何学会调用外部工具的？

#### 核心问题

这一篇在回答：

> 工具调用能力是模型天然就有的吗？

#### 我的笔记

- 不是天然就有的
- 预训练模型擅长的是“预测下一个 token”
- 这能带来语言理解和文本生成能力
- 但不会天然带来严格的工具调用能力

也就是说，模型会“说自己可以查天气”，不代表它会稳定输出：

- 工具名
- 参数 JSON
- 符合约束的结构化字段

这篇重点是把“语言能力”和“工具调用能力”分开。

#### 我觉得这篇最该记住的点

- 预训练只保证通用语言能力
- 工具调用需要专门后训练
- 不训练，模型就容易只会“描述意图”，不会“规范调用”

#### 面试回答关键词

预训练、token 预测、语言能力、工具调用能力、后训练

### 3. 大模型的 Function Call 能力是怎么训练出来的？

#### 核心问题

这一篇在回答：

> 如果工具调用不是天然具备的，那它具体靠什么训练出来？

#### 我的笔记

我把这篇总结成两句话：

- `SFT` 解决“会不会调”
- `RLHF / RLAIF` 解决“该不该调”

更具体地说：

##### 3.1 SFT 阶段

训练样本会告诉模型：

- 当前有哪些工具
- 用户的问题是什么
- 正确的工具调用长什么样
- 工具返回什么结果
- 最终答案怎么写

模型通过这些样本学会：

- 识别什么时候需要工具
- 怎么按 `schema` 填参数
- 怎么输出规范结构

##### 3.2 偏好优化阶段

只会“调用”还不够，还要知道“什么时候不该调用”。

例如：

- `1 + 1` 这种问题直接回答就行
- “今天北京天气如何”更适合调用实时工具

所以偏好优化阶段会进一步塑造模型的调用边界感。

#### 我觉得这篇最该记住的点

- 工具调用是训练出来的，不是自动涌现的
- `SFT` 重在格式和流程
- `RLHF / RLAIF` 重在策略和边界

#### 面试回答关键词

SFT、RLHF、RLAIF、结构化输出、调用时机、边界判断

## 三、MCP 协议篇

### 4. 什么是 MCP？讲讲它的核心内容？

#### 核心问题

这一篇在回答：

> 当工具越来越多以后，怎么把工具统一接给模型？

#### 我的笔记

MCP 可以理解成：

> **面向 AI 应用和工具服务之间的标准化接入协议。**

它关注的不是单次“调用哪个函数”，而是更上层的问题：

- 工具如何统一暴露
- 工具如何被发现
- 工具如何标准化接入
- AI 客户端如何和工具服务协同

这篇里最重要的不是记死定义，而是理解 MCP 出现的背景：

- 工具多了
- 接入方式乱了
- 每个 AI 应用都重复造轮子

所以需要标准化。

#### 我觉得这篇最该记住的点

- MCP 的重点是 **接入标准化**
- 它发生在 AI Client 和 MCP Server 之间
- 它不只关心“调一次”，还关心“怎么接一整批工具”

#### 面试回答关键词

标准化接入、工具发现、AI Client、MCP Server、统一协议

### 5. MCP 和 Function Calling 有什么区别？

#### 核心问题

这一篇在回答：

> MCP 和 Function Calling 看起来都像在“让模型用工具”，那它们到底差在哪？

#### 我的笔记

我对两者的区分方式是：

- `Function Calling`：解决“这次调哪个函数、参数是什么”
- `MCP`：解决“工具怎么统一接入、统一发现、统一暴露”

所以：

- Function Calling 偏 **单次调用**
- MCP 偏 **工具接入层**

这篇文章的关键价值，在于防止把两者误当成替代关系。

#### 我觉得这篇最该记住的点

- Function Calling 是更底层的调用表达机制
- MCP 是更上层的工具接入规范
- 一套系统里二者可以同时存在

#### 面试回答关键词

单次调用、工具接入、分层、非替代关系

---

### 6. MCP 协议通常采用什么通信方式？

#### 核心问题

这一篇在回答：

> MCP 作为协议，底层消息一般怎么传？

#### 我的笔记

MCP 作为工具接入协议，实际落地时会涉及不同消息传输方式。  
重点不是背协议名，而是理解它为什么要讨论通信：

- 工具调用不是一次性静态请求
- 可能需要流式返回
- 可能需要持续会话
- 可能要在客户端和服务端之间维持上下文

所以这篇本质是在为后面的 `SSE / WebSocket / WebRTC` 做铺垫。

#### 我觉得这篇最该记住的点

- 协议设计离不开底层传输方式
- 不同通信方式决定实时性、交互性和复杂度
- MCP 是“上层语义”，底层还得靠具体传输机制承载

#### 面试回答关键词

协议语义、消息传输、实时性、上下文同步、通信承载

## 四、通信协议补充篇

### 7. WebSocket 和 SSE 通信的区别及局限性？

#### 核心问题

这一篇在回答：

> 在 AI 场景里，为什么有时用 SSE，有时又要用 WebSocket？

#### 我的笔记

我把两者区别记成：

- `SSE`：服务端单向推送
- `WebSocket`：客户端和服务端双向通信

如果只是做：

- 模型流式输出
- 服务端持续推 token

那 `SSE` 就很轻量、很好用。

如果场景变成：

- 前后端都要主动发消息
- 要做状态同步
- 要做更复杂的会话交互

那 `WebSocket` 更合适。

局限性方面可以简单记：

- SSE 简单但偏单向
- WebSocket 更灵活但复杂度更高

#### 我觉得这篇最该记住的点

- SSE 适合流式输出
- WebSocket 适合双向交互
- 选型本质是看交互模式，不是看哪个更“高级”

#### 面试回答关键词

单向推送、双向连接、流式输出、复杂交互、选型场景

---

### 8. WebSocket、SSE 和 WebRTC 的区别是什么？

#### 核心问题

这一篇在回答：

> 如果再把 WebRTC 加进来，这三种协议各适合什么场景？

#### 我的笔记

我会这样记：

- `SSE`：最适合服务端流式推送文本
- `WebSocket`：最适合通用双向交互
- `WebRTC`：更偏低延迟实时媒体或实时通信

AI 应用里常见情况是：

- LLM 文本流式输出：优先想到 `SSE`
- 聊天协作、控制指令双向收发：优先想到 `WebSocket`
- 实时音视频 Agent、语音互动：更可能涉及 `WebRTC`

#### 我觉得这篇最该记住的点

- 不是所有实时场景都该上 WebRTC
- 不是所有流式场景都需要 WebSocket
- 协议选择要和交互数据类型、方向性、延迟要求匹配

#### 面试回答关键词

文本流、双向交互、低延迟实时通信、音视频、协议选型

## 五、协议对比与选型篇

### 9. 为什么有了 Function Calling，还需要 MCP？

#### 核心问题

这一篇在回答：

> 既然 Function Calling 已经能让模型调工具了，为什么还要再搞一个 MCP？

#### 我的笔记

这篇的重点是看规模变化：

- 工具少时，Function Calling 已经够用
- 工具多、团队多、应用多时，仅靠 Function Calling 不够

因为 Function Calling 更像：

- “模型该怎么调用工具”

而 MCP 解决的是：

- “工具如何被多个 AI 应用统一接入和复用”

换句话说：

> Function Calling 解决的是“调用表达”，MCP 解决的是“工具生态管理”。

#### 我觉得这篇最该记住的点

- 工具少时，FC 足够简单直接
- 工具规模化后，MCP 的统一接入价值才显现
- MCP 的出现是工程复杂度上升后的产物

#### 面试回答关键词

规模化、复用、标准化、生态管理、复杂度上升

---

### 10. MCP 和 Function Calling 在实际项目中该怎么选型？

#### 核心问题

这一篇在回答：

> 真实项目里，什么时候选 Function Calling，什么时候选 MCP？

#### 我的笔记

我把选型策略记成一句话：

> **简单项目先 FC，复杂工具生态再考虑 MCP。**

大致可以这样理解：

##### 更适合 Function Calling 的情况

- 工具数量不多
- 单应用内部使用
- 追求实现简单
- 更关注调用本身而不是工具治理

##### 更适合 MCP 的情况

- 工具数量多
- 多 Agent / 多应用共享工具
- 需要统一暴露和发现
- 需要更标准化的扩展能力

#### 我觉得这篇最该记住的点

- 选型不是技术优劣，而是工程阶段问题
- FC 更轻，MCP 更体系化
- 先用简单方案跑通，再按复杂度升级，是更务实的路线

#### 面试回答关键词

轻量、复杂生态、共享工具、标准化、工程阶段

---

### 11. 为什么推理模型通常不支持 MCP，却可以支持 Function Calling？

#### 核心问题

这一篇在回答：

> 为什么很多推理模型能支持 Function Calling，却不直接支持 MCP？

#### 我的笔记

我对这篇的理解是：

- Function Calling 更贴近模型本身的输出能力
- 它只要求模型会输出结构化调用请求

而 MCP 不只是模型能力问题，它还要求：

- 客户端具备 MCP 协议能力
- 工具端按 MCP 规范暴露能力
- 整个系统围绕 MCP 做集成

所以，模型“支持 FC”比较自然，因为它本质上还是文本/结构化输出问题；  
而“支持 MCP”往往是系统级集成问题，不只是模型本身的问题。

#### 我觉得这篇最该记住的点

- FC 更像模型能力
- MCP 更像系统集成能力
- 推理模型本身不需要理解完整 MCP 生态，也能支持 FC

#### 面试回答关键词

模型输出能力、系统集成、协议栈、客户端支持、生态支持

## 六、多 Agent 协作篇

### 12. 什么是 A2A 协议？它和 MCP 协议的区别是什么？

#### 核心问题

这一篇在回答：

> 当系统里不止一个 Agent 时，Agent 和 Agent 之间怎么协作？

#### 我的笔记

我会这样区分：

- `MCP`：偏工具接入协议
- `A2A`：偏 Agent 与 Agent 的协作协议

MCP 关心的是：

- 工具怎么暴露
- AI 客户端怎么接工具

A2A 关心的是：

- 任务怎么在 Agent 之间转交
- 状态怎么同步
- 多 Agent 怎么协同完成目标

#### 我觉得这篇最该记住的点

- MCP 面向“Agent/客户端 ↔ 工具”
- A2A 面向“Agent ↔ Agent”
- 两者解决对象不同，层次也不同

#### 面试回答关键词

Agent 协作、任务委派、状态流转、工具接入、协议分工

---

### 13. Skill 是什么？为什么它不是 Function？

#### 核心问题

这一篇在回答：

> Skill 看起来也像功能，那为什么不能把它直接理解成函数？

#### 我的笔记

我对 Skill 的理解是：

> **Skill 是 Agent 对外声明的能力标签，而 Function 是具体执行动作。**

例如：

- `get_weather(city)` 是 Function
- “竞品分析”“代码审查”“生成周报”更像 Skill

Function 粒度更细，Skill 粒度更粗。  
一个 Skill 背后可能包含：

- 多轮推理
- 多次工具调用
- 访问多个数据源
- 调用其他 Agent

#### 我觉得这篇最该记住的点

- Skill 是能力描述，不是原子调用
- Function 偏执行，Skill 偏任务承接能力
- Skill 可以由多个 Function 组合实现

#### 面试回答关键词

能力标签、原子操作、粗粒度、任务级抽象、组合实现

---

### 14. MCP 和 Agent Skill 的区别是什么？

#### 核心问题

这一篇在回答：

> MCP 和 Skill 都像“给 Agent 加能力”，它们边界到底在哪？

#### 我的笔记

我的理解是：

- `MCP` 解决的是外部工具如何标准化接入
- `Skill` 解决的是 Agent 自己对外声明“我能做什么”

也就是说：

- MCP 更像“工具层协议”
- Skill 更像“能力层抽象”

Skill 可以内部调用 MCP 暴露出来的工具，但 Skill 本身不是工具协议。

#### 我觉得这篇最该记住的点

- MCP 是接工具
- Skill 是表能力
- Skill 之下可以使用 MCP 接进来的工具

#### 面试回答关键词

工具层、能力层、对外声明、内部实现、分层抽象

---

### 15. Function Calling、Agent Skill、MCP 三者的区别是什么？

#### 核心问题

这一篇在回答：

> 到底该怎么把 FC、Skill、MCP 放到同一张图里理解？

#### 我的笔记

这是整套专题里最值得总结的一篇。

我把三者关系写成一句话：

> **Function Calling 负责“怎么调一次函数”，MCP 负责“工具怎么标准化接入”，Skill 负责“哪个 Agent 擅长处理什么任务”。**

如果按层次看：

| 层次 | 解决的问题 | 对象 |
| --- | --- | --- |
| Skill 层 | 任务该交给谁 | Agent |
| MCP 层 | 工具怎么接进来 | AI Client / Tool Server |
| Function Calling 层 | 模型怎么发起一次调用 | LLM / 宿主程序 |

#### 我觉得这篇最该记住的点

- 三者不是竞争关系
- 三者是上下分层关系
- 上层能力依赖下层能力

#### 面试回答关键词

分层、调用层、接入层、能力层、非替代关系

## 七、工程治理篇

### 16. LLM 网关层解决了什么问题？

#### 核心问题

这一篇在回答：

> 当模型、工具、团队、供应商都多起来以后，系统怎么治理？

#### 我的笔记

LLM Gateway 不只是反向代理，它更像 AI 系统的治理中枢。

我把它解决的问题概括成 6 类：

1. **统一入口**：业务只接一个模型入口
2. **模型路由**：底层可接 OpenAI、Claude、自建模型等不同供应商
3. **Key 管理**：集中保管真实密钥，减少泄漏风险
4. **限流配额**：避免资源被某个团队或应用打满
5. **成本追踪**：统计 token 消耗和费用
6. **安全审计**：做权限、过滤、审计和风险控制

为什么它重要？  
因为到了生产环境，真正难的往往不再是“能不能调通”，而是：

- 怎么稳定
- 怎么控成本
- 怎么统一接入
- 怎么防风险

#### 我觉得这篇最该记住的点

- LLM Gateway 的重点是治理，不只是转发
- 它是从 demo 走向生产环境时最容易被低估的一层
- 多模型时代几乎一定会需要它

#### 面试回答关键词

统一入口、模型路由、Key 管理、配额、成本、安全审计

## 八、我对整套专题的理解

学完这 16 篇后，我对整套专题的理解可以压缩成下面四层：

### 1. 单次调用层

由 `Function Calling` 负责。

它解决的是：

> 模型如何表达一次结构化工具调用请求。

### 2. 工具接入层

由 `MCP` 负责。

它解决的是：

> 工具如何被统一接入、统一发现、统一暴露。

### 3. 协作调度层

由 `Skill` 和 `A2A` 负责。

它解决的是：

> 多 Agent 系统里，任务该给谁，Agent 之间怎么协作。

### 4. 生产治理层

由 `LLM Gateway` 负责。

它解决的是：

> 多模型、多团队、多成本场景下，怎么统一治理。

所以我最后的总理解是：

> 这套专题不是在讲一个单独技术点，而是在讲“AI 系统从单工具调用到规模化工程落地”的完整演进路径。

## 九、面试速记版

### 1. 必背判断句

- 模型不会直接执行工具，它只输出调用请求
- Function Calling 解决的是一次调用如何表达
- 工具调用能力来自后训练，不是预训练天然具备
- `SFT` 解决“会不会调”，偏好优化解决“该不该调”
- MCP 解决的是工具标准化接入
- A2A 解决的是 Agent 和 Agent 的协作
- Skill 是能力声明，不是函数本身
- LLM Gateway 解决的是统一接入、成本、安全和治理

### 2. 一段话总结全专题

如果让我用一段话总结这套工具调用专题，我会这样说：

> 我会把这套体系分成四层。最底层是 Function Calling，它让模型输出结构化工具调用请求；再上一层是 MCP，它让工具可以被标准化接入和发现；再上一层是 Skill 和 A2A，它们让多 Agent 系统可以声明能力并进行协作；真正上线后，还需要 LLM Gateway 做统一接入、路由、Key 管理、配额、成本和安全治理。这样整套 AI 工具调用体系才算完整。

### 3. 复习顺序建议

如果后面继续复习，我会这样回顾：

1. 先复习 1~3，打牢 Function Calling 和训练基础
2. 再复习 4~6，理解 MCP 的位置
3. 再看 7~8，把通信协议区别理顺
4. 再看 9~11，形成选型和边界感
5. 最后看 12~16，串起多 Agent 与工程治理
