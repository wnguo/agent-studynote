# 一、先建立全局认知

## 1. 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）本质上是：

- 先从外部知识源中做检索
- 再把检索结果作为上下文交给 LLM 生成答案

它不是：

- 单纯搜索
- 单纯大模型裸答
- 单纯微调

它更像是给大模型加上一个“开卷考试”机制：

- 搜索负责找资料
- 模型负责读资料、组织答案、生成表达

### 一句话理解

- 搜索决定“能不能找到”
- 大模型决定“能不能答得好”

## 2. 为什么需要 RAG？

RAG 主要解决 3 类问题：

### 2.1 知识时效性

- 大模型训练完成后，知识就被冻结了
- 新政策、新产品、新版本信息可能不在参数里

### 2.2 私有知识覆盖

- 企业内部文档、SOP、知识库、客服规则、研发规范，通常不在预训练语料中
- 单靠基础模型无法回答组织内部知识

### 2.3 幻觉问题

- 当模型没有足够依据时，容易“看起来合理地瞎编”
- RAG 的核心价值之一，就是给回答提供“外部证据”

> [!WARNING]
> RAG 不是彻底消灭幻觉，而是通过“先找证据，再回答”来显著降低幻觉概率。

### 2.4 根本原因

这几个问题的共同根源是：

- 知识被固化在模型参数中
- 推理时如果没有外部知识输入，模型只能依赖参数记忆

**面试高频考点**

- 为什么大模型需要 RAG？
  - 参数知识有截止时间
  - 私有知识不在训练集中
  - 裸答容易幻觉
  - 需要可追溯、可引用的答案能力

## 3. RAG 和微调的区别

### 3.1 核心区别

- 微调：修改模型参数
- RAG：不改参数，在推理时动态注入知识

### 3.2 各自适用场景

#### 更适合 RAG

- 知识变化快
- 私有知识多
- 需要可引用来源
- 需要持续动态更新知识库

#### 更适合微调

- 需要固定输出风格
- 需要特定格式能力
- 需要让模型更擅长某类任务范式
- 需要稳定行为偏好

### 3.3 最佳实践

真实业务通常不是二选一，而是：

- 微调负责“行为和风格”
- RAG 负责“知识供给”

> [!TIP]
> 一个非常实用的面试表达是：微调解决“怎么答”，RAG 解决“答什么”。

**面试高频考点**

- RAG 和微调怎么选？
  - 知识变动频繁：优先 RAG
  - 风格/格式要求强：考虑微调
  - 生产常见方案：二者结合

# 二、RAG 的完整工作流程

RAG 通常分为两个阶段：

- 离线阶段：建知识库
- 在线阶段：回答问题

## 1. 离线阶段：先把知识准备好

### 1.1 文档加载

常见数据源包括：

- PDF
- Word
- Markdown
- 网页
- FAQ
- 数据库记录
- 内部知识平台内容

### 1.2 文档切割（Chunking）

整篇文档通常不能直接入库，因为：

- Embedding 模型有输入长度限制
- 一整篇压成一个向量会丢掉局部语义

所以要把文档切成多个 Chunk。

### 1.3 向量化（Embedding）

把每个 chunk 转成向量，核心前提是：

- 语义相似的文本，在向量空间更接近

### 1.4 入库

每条记录通常至少包含：

- `vector`：用于相似度检索
- `text`：检索命中后真正给模型读的原文
- `metadata`：来源、标题、页码、标签、权限等

## 2. 在线阶段：回答用户问题

在线链路一般是：

1. 用户提问
2. Query Rewrite
3. Query 向量化
4. 粗排召回
5. Rerank 精排
6. Prompt 拼装
7. LLM 生成答案
8. 返回答案与来源

### 2.1 为什么不能直接把问题扔给模型？

- 上下文窗口有限
- 知识库太大，塞不进去
- 裸答容易幻觉
- 用户问题表达常常不规范，不利于直接检索

### 2.2 粗排 vs 精排

- 粗排：追求快，先从海量内容中捞候选
- 精排：追求准，把真正相关的排到前面

常见工程实践：

- 粗排取 Top-20 / Top-30
- 精排后保留 Top-3 / Top-5 给模型

**面试高频考点**

- 一条完整的 RAG 在线链路是什么？
  - 用户问题 → Query 处理 → 检索 → 多路召回 → Rerank → Prompt → LLM 生成 → 返回引用

# 三、Chunking：RAG 成败的第一道坎

## 1. 为什么要切块？

如果不切块，会有几个问题：

- 太长，模型或 embedding 模型吃不下
- 太大，语义被平均化
- 太粗，检索定位不准

### 核心理解

Chunking 决定了：

- 资料会被怎样切分
- 检索能否精准命中
- 模型最终读到的是不是合适的上下文

> [!WARNING]
> 很多 RAG 效果差，不是模型差，而是 chunk 切坏了。

## 2. Chunk 存储结构

一个 chunk 通常包含：

- 向量
- 原文
- metadata

分别承担的作用：

- 向量：负责“找”
- 原文：负责“读”
- metadata：负责“筛选、权限、来源追踪”

## 3. Chunk 粒度怎么定？

常见经验起点：

- `500 ~ 1000 token`
- 加一定 `overlap`

但真正关键不在于死记某个数字，而在于：

> [!IMPORTANT]
> Chunk 的大小应当由文档类型、内容结构、检索目标共同决定。

例如：

- FAQ：可以更小
- 长文说明书：可按段落或标题切
- 代码：按函数/类切
- 法规：按条款切

## 4. 常见 Chunking 策略

### 4.1 固定大小 + overlap

适合：

- 通用场景
- 快速起步
- 原型验证

优点：

- 简单
- 稳定
- 易实现

缺点：

- 容易切断语义边界

### 4.2 按语义边界切

比如按：

- 句子
- 段落
- 标题层级

优点：

- 语义完整性更强

### 4.3 父子切割

- 小块负责检索
- 大块负责给模型阅读

本质：

- 小块提召回精度
- 大块保上下文完整

### 4.4 代码按结构切

代码文档适合按：

- 函数
- 类
- 模块
- 文件结构

### 4.5 命题化切割

把内容拆成独立“命题”进行存储和检索。

优点：

- 更利于问答
- 语义单位更明确

缺点：

- 成本更高
- 实现更复杂

**面试高频考点**

- Chunk 太大和太小分别有什么问题？
  - 太大：检索不精确，噪音高
  - 太小：上下文不完整，语义容易断裂

# 四、如何规避“语义被切断”？

这个问题的本质是：

- chunk 单独拿出来后，失去上下文
- 检索虽然命中了碎片，但模型无法准确理解

## 1. 切的时候别截断

常见手段：

- `overlap`
- 按标题/段落等语义边界切割

## 2. 切完后补上下文

常见手段：

- 句子窗口检索
- 父子切割
- 命题化切割
- Contextual Retrieval

## 3. Contextual Retrieval

Contextual Retrieval 是一个很值得记的高级思路：

- 在做 embedding 前
- 先让模型根据全文，为每个 chunk 补一段背景说明
- 再把“背景说明 + chunk”一起向量化

这样做的意义是：

- 让孤立 chunk 带上“它在整篇文档中的语境”

适用场景：

- 上下文依赖很强的文档
- chunk 单独看容易看不懂
- 需要更高质量召回

代价：

- 有额外的 LLM 成本

> [!TIP]
> 如果你的知识库内容经常是“这段话脱离上下文就看不懂”，那就该认真考虑 Contextual Retrieval。

**面试高频考点**

- 怎么缓解 chunk 语义断裂？
  - overlap
  - 语义边界切割
  - 父子切割
  - 句子窗口
  - Contextual Retrieval

# 五、Embedding：检索质量的基础

## 1. Embedding 是什么？

Embedding 的本质是：

- 把文本映射到高维向量空间

在 RAG 里，它的作用是：

- 不按字面匹配
- 而按语义相似度匹配

## 2. 选 Embedding 模型看什么？

### 2.1 中文能力

中文业务优先关注中文效果。  
笔记中提到中文常用可关注 `BGE` 系列。

### 2.2 向量维度

- 维度高：通常表达能力更强
- 但也更占存储和算力

### 2.3 最大输入长度

决定单个 chunk 最长可以到什么程度。

> [!NOTE]
> 选 embedding 模型，不只是看排行榜，更要看：
> - 你的语言场景
> - 文本长度
> - 成本预算
> - 线上延迟要求

## 3. 怎么评估 Embedding 模型？

不要只看通用榜单，要在自己的业务数据上测。

因为：

- 通用 benchmark 不等于你的业务效果
- 医疗、法律、客服、研发文档差异很大

关注指标：

- 召回率
- 排名质量
- 最终问答表现

## 4. Embedding 算法演进

### 第一代：静态词向量

代表：

- Word2Vec
- GloVe

特点：

- 一个词一个固定向量
- 无法处理上下文和多义词

### 第二代：上下文相关表示

代表：

- BERT

特点：

- 同一个词在不同上下文中表示不同
- 表达能力更强

不足：

- 不一定适合大规模离线检索场景

### 第三代：句子级对比学习 Embedding

代表：

- SBERT
- SimCSE
- BGE

特点：

- 更适合句子/段落相似度
- 更适合 RAG 文档预编码
- 是当前主流路线

**面试高频考点**

- Embedding 模型怎么选？
  - 先看语种
  - 再看长度
  - 再看维度成本
  - 最后一定要做业务数据评测

# 六、向量数据库：RAG 的检索底座

## 1. 什么是向量数据库？

向量数据库是专门用于：

- 存储高维向量
- 做近似最近邻搜索（ANN）

的数据库系统。

为什么不能只用普通数据库？

- 传统索引擅长结构化字段精确匹配
- 高维向量相似度搜索是另一类问题

## 2. 常见选型思路

### 本地开发 / Demo

- `Chroma`
- 优点：简单、零配置、适合实验

### 中小规模生产

- `Qdrant`
- API 简洁，体验好

### 大规模 / 分布式

- `Milvus`
- 更适合百万级以上和高并发

### 托管服务

- `Pinecone`
- 运维成本低

### PostgreSQL 体系

- `pgvector`
- 如果已有 PG 基建，可以快速接入

> [!TIP]
> 选型的本质不是“谁最火”，而是：
> - 数据量级
> - 并发要求
> - 运维能力
> - 团队技术栈

## 3. 向量库实战案例（Milvus）

专题里给出的工程案例可以重点记忆：

- 生产使用：`Milvus`
- 数据量级：约百万条向量
- 向量维度：`1024`
- 索引：`HNSW`
- 查询延迟：约 `20~50ms`

选择 `Milvus` 的原因：

- 支持分布式
- 支持读写分离
- 更适合大规模生产环境

## 4. 真实瓶颈

### 4.1 内存压力

百万级 + 高维向量，会显著占用内存。  
优化方式：

- `SQ8` 标量量化
- 把 `float32` 压缩成 `int8`
- 降低内存开销

### 4.2 批量写入影响查询

后台 `Segment` 合并可能带来查询抖动。

解决思路：

- 低峰期写入
- 小批次分批更新
- 降低写入和查询资源争抢

**面试高频考点**

- 为什么生产用 Milvus，而不是只用 Chroma？
  - Chroma 更偏本地开发和实验
  - Milvus 更适合百万级、分布式、高并发

# 七、检索：不只是“向量搜一下”

## 1. 向量检索 vs 关键词检索

### 关键词检索（如 BM25）

优势：

- 精确命中强
- 对数字、专有词、型号、缩写友好

不足：

- 不理解语义近义表达

### 向量检索

优势：

- 语义理解更强
- 对同义改写更友好

不足：

- 对精确关键词、缩写、ID 可能不稳定

### 结论

真实系统通常采用：

- 向量检索补语义
- BM25 补精确匹配
- 再做融合

> [!IMPORTANT]
> 单一路径检索很容易有盲区，混合检索通常更稳。

## 2. Query Rewrite：为什么要改写用户问题？

用户的原始提问通常具有这些特点：

- 口语化
- 省略信息
- 带指代
- 有缩写
- 和文档书面表达差异大

因此需要在检索前对 query 做增强。

### 常见方法

#### 2.1 直接改写

把问题改得更完整、更标准。

#### 2.2 Query 扩展

补充：

- 同义词
- 相关术语
- 上下位概念

#### 2.3 HyDE

先让 LLM 生成一个“假设答案”，再拿假设答案去检索。

优点：

- 改善语义桥接

风险：

- 假设偏了，会把召回带偏

#### 2.4 Step-back Prompting

先把问题抽象成更通用的问题，再做背景召回。

## 3. 多路召回

多路召回的核心思想是：

> 不相信单一路径足够好，而是让多种方式并行召回，再做融合。

常见组合：

- 向量检索
- `BM25`
- 多 Query 召回

常见融合方法：

- `RRF`（Reciprocal Rank Fusion）

## 4. Rerank：效果最稳的提升手段之一

Rerank 的职责是：

- 在粗排召回的候选中
- 精确挑出真正相关的内容

典型做法：

- 先粗排 Top-20 / Top-30
- 再 Rerank
- 最后保留 Top-3 / Top-5 进入 Prompt

为什么它重要？

- 粗排追求快，不够准
- 直接把粗排结果喂给 LLM，会带入噪音

**面试高频考点**

- 为什么 Rerank 很重要？
  - 因为“召回到”不等于“排得对”，而模型读上下文的预算非常宝贵

# 八、RAG 检索优化的四层框架

这个四层框架非常适合背：

## 1. 索引层：知识怎么存

关注：

- Chunking
- overlap
- metadata 设计
- 父子切割
- Contextual Retrieval

## 2. 查询层：问题怎么转

关注：

- Query Rewrite
- Query 扩展
- `HyDE`
- Step-back Prompting

## 3. 召回层：从哪几条路找

关注：

- 向量检索
- `BM25`
- 混合检索
- 多路召回
- `RRF`

## 4. 重排序层：谁最相关

关注：

- Rerank
- 候选压缩
- Top-N 控制

> [!IMPORTANT]
> 检索效果不好时，别只盯着向量库。你需要按这四层逐层排查问题到底出在哪。

# 九、更复杂的 RAG 范式

## 1. 三代演进

### 1.1 Naive RAG

流程最基础：

- 检索
- 拼 Prompt
- 回答

优点：

- 简单
- 好落地

缺点：

- 复杂问题处理能力弱

### 1.2 Advanced RAG

在 Naive 基础上增加：

- Query 改写
- 混合检索
- Rerank

### 1.3 Modular RAG

把每个组件模块化，按场景自由组合。

## 2. 典型高级范式

### 2.1 Self-RAG

核心思想：

- 模型自己判断要不要检索
- 自己评估检索结果是否够用

### 2.2 CRAG

核心思想：

- 如果知识库召回效果差
- 切换到其他外部来源做兜底

### 2.3 GraphRAG

核心思想：

- 先建图谱，再做实体关系级推理

适合：

- 多跳推理
- 跨实体关系分析

### 2.4 Agentic RAG

核心思想：

- 让 RAG 进入多轮推理和动态检索循环

**面试高频考点**

- Naive RAG、Advanced RAG、Modular RAG 的区别是什么？
  - Naive：最简单
  - Advanced：增加检索优化
  - Modular：组件解耦、可灵活编排

# 十、什么时候该引入图数据库？

如果你的问题不是“找相似段落”，而是“找实体之间的关系”，那就要认真考虑图数据库。

适用场景：

- 企业关系图谱
- 医疗知识图谱
- 法规关联
- 代码依赖分析
- 供应链上下游分析

> [!TIP]
> 一个很好记的判断方式：问题核心在“关系”，就考虑图；问题核心在“相似文本”，优先检索。

# 十一、如何规避 RAG 幻觉？

## 1. 没召回到相关内容

后果：

- 模型没资料
- 只能靠参数记忆甚至瞎编

## 2. 召回了，但模型没老实照着答

后果：

- 模型在有依据的情况下，仍然做超范围推断

## 3. 常见治理手段

### 3.1 Prompt 强约束

明确要求：

- 只能根据资料回答
- 没依据就说不知道

### 3.2 检索质量门控

- 如果 Rerank 分数低于阈值
- 拒答、追问或兜底，而不是硬答

### 3.3 引用核查

- 关键结论必须对应到具体 chunk

### 3.4 结构化输出 + 溯源

- 让回答附带来源编号

> [!IMPORTANT]
> 低成本高收益组合：
> - Prompt 强约束
> - 检索质量门控

**面试高频考点**

- RAG 幻觉怎么控制？
  - 提高检索质量
  - 限制模型超范围发挥
  - 做证据引用
  - 做低分拒答/兜底

# 十二、如何量化 RAG 效果？

## 1. 检索层评估

### Hit@K

问的是：

- 正确内容是否出现在前 K 个结果中？

### MRR

问的是：

- 正确结果排第几？

理解：

- Hit@K 高但 MRR 低，说明“找到了，但排得不够前”

## 2. 生成层评估

常见可关注：

### Faithfulness

- 回答是否忠于上下文

### Answer Relevancy

- 回答是否真正回答了问题

### Context Recall

- 检索上下文是否足以支撑回答

## 3. 线上指标同样关键

最终要看业务结果，例如：

- 点踩率
- 追问率
- 转人工率
- 用户满意度

> [!TIP]
> 离线指标帮助“定位问题”，线上指标帮助“验证价值”。

**面试高频考点**

- 为什么只看最终回答好不好还不够？
  - 因为你需要拆清楚问题到底出在检索没找到、排序没排对，还是模型生成没控制好

# 十三、知识库如何动态更新？

生产环境一定会遇到：

- 文档新增
- 文档修改
- 文档删除

核心要求：

- 增量更新
- 不要每次都全量重建

## 1. 常用更新流程

1. 检测文档变化
2. 删除旧 chunk / 旧向量
3. 重新切割
4. 重新向量化
5. 写回向量库

一句话总结：

> [!NOTE]
> 动态更新的核心是“先删后增”，而不是“每次全量重建”。

## 2. 怎么知道文档变了？

常见方法：

- 对文档内容做 `hash`
- 维护文档 ID 和 chunk 的映射关系
- 对比新旧 hash

如果 hash 变了：

- 说明内容发生变化
- 触发增量更新流程

## 3. 两种变更感知方式

### 3.1 Polling

适合：

- 更新不频繁
- 实时性要求不高

优点：

- 简单

缺点：

- 有延迟
- 扫描成本高

### 3.2 Event-Driven

例如：

- Kafka
- RabbitMQ
- Webhook

适合：

- 高实时要求
- 频繁变更场景

优点：

- 更新及时

缺点：

- 架构更复杂

**面试高频考点**

- 知识库更新为什么不能总做全量重建？
  - 成本高
  - 时效差
  - 对大规模数据不现实

# 十四、RAG 真正难的地方

> [!WARNING]
> 最难的不是“把 RAG 跑起来”，而是“把 RAG 调好”。

主要难点有 3 个：

## 1. 文档预处理

- PDF、表格、图片、复杂版式解析困难
- 输入质量差会直接影响检索效果

## 2. 检索质量调优

- Chunking
- Embedding
- 检索策略
- Query Rewrite
- Rerank

这些环节互相影响，调优难度很高。

## 3. 效果评估

- 很多时候不知道到底是检索错了还是生成错了
- 没有评估体系，优化容易盲目

**面试高频考点**

- RAG 最难的部分是什么？
  - 文档预处理
  - 检索链路调优
  - 评估与问题定位

# 十五、面试回答框架

如果面试官问：“你做过 RAG 吗？”可以按下面顺序回答：

## 1. 先讲目标

- 解决知识时效性
- 解决私有知识接入
- 降低幻觉

## 2. 再讲完整链路

- 离线：加载文档 → Chunking → Embedding → 入库
- 在线：Query 改写 → 检索 → 多路召回 → Rerank → Prompt → 生成

## 3. 再讲关键选型

- Embedding 模型怎么选
- 向量库怎么选
- chunk 粒度怎么定
- 是否做混合检索和精排

## 4. 再讲生产问题

- 幻觉怎么控制
- 效果怎么评估
- 知识库怎么动态更新
- 性能瓶颈怎么处理

## 5. 最后讲真实工程经验

例如：

- `Milvus`
- 百万级向量
- `HNSW`
- 查询延迟 `20~50ms`
- `SQ8` 压缩内存
- 错峰写入减少 Segment 合并抖动

> [!IMPORTANT]
> 面试里最加分的，不是讲概念，而是讲：
> - 数据量级
> - 性能指标
> - 踩过什么坑
> - 怎么优化的

# 十六、最终总结

如果把整个专题压成一句话：

> RAG 不是“检索一下再让模型回答”这么简单，而是一整套围绕知识供给、检索质量、生成可信度和线上可运维性构建的工程体系。

真正需要掌握的 4 条主线：

1. **知识怎么组织**：文档加载、Chunking、Embedding、向量数据库
2. **内容怎么召回**：向量检索、BM25、Query Rewrite、多路召回、Rerank
3. **答案怎么可信**：引用、门控、幻觉控制、评估指标
4. **系统怎么上线**：动态更新、性能优化、监控、兜底策略
